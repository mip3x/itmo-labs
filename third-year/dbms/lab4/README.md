# Лабораторная работа №4

## Условие. Вариант 1

### Этап 1. Развертывание и топология кластера

1. Разверните кластер Cassandra из 3-х узлов (cassandra-1, cassandra-2, cassandra-3). Выберите подходящий RF, обоснуйте выбор.

2. Настройте cassandra-1 в качестве seed-узла.

3. После запуска кластера докажите, что cassandra-1 работает в качестве seed-узла.

4. Покажите, что все узлы перешли в состояние UN.

### Этап 2. Согласованность данных

1. Создайте таблицу sensors (sensor_id, time, val). PK: sensor_id. В качестве CL установите: QUORUM.

2. Добавьте 10 записей в созданную таблицу. Определите физическое расположение данных, их распределение по узлам, приведите статистику.

3. Экспериментально определите предел отказоустойчивости кластера при CL=QUORUM. Зафиксируйте поведение при последовательном отключении узлов.

4. Что изменится при использовании CL=ONE? Продемонстрируйте результат.

### Этап 3. Работа с данными

1. Удалите 10% записей (с наименьшим time). Изменится ли объем данных на диске и как? Объясните почему.

2. Идентифицируйте записи-надгробия (tombstones) в файлах данных. Опишите, как это сделать и почему они появились.

3. Поменяйте значение RF (сделайте RF = 1). Изменится ли объем данных на диске и как? Объясните почему. Произведите cleanup узлов

### Этап 4. Шардирование

1. Создайте ситуацию, при которой один из узлов (cassandra-2) станет «горячей точкой», приняв на себя аномально высокий объем данных. Подтвердите перекос через статистику распределения ключей.

2. Перепроектируйте первичный ключ, внедрив Composite Partition Key, чтобы распределить данные по разным физическим узлам и устранить перекос.

3. Докажите, что после изменения модели нагрузка распределяется равномерно.
